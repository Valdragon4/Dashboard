# Story 1.4: Connecteur BoursoBank

## Status

Done

## Story

**As a** utilisateur,
**I want** pouvoir connecter mes comptes BoursoBank pour synchronisation automatique,
**so that** je n'ai plus besoin d'importer manuellement les CSV.

## Acceptance Criteria

1. Recherche et évaluation des options techniques disponibles :
   - Exploration de l'API/scraping HTTP direct (si disponible)
   - Vérification de l'API PSD2 officielle BoursoBank
   - Évaluation de Selenium/Playwright comme fallback
   - Documentation des options dans `docs/connectors/boursorama-research.md`
2. Classe `BoursoBankConnector` créée avec implémentation de l'authentification selon l'approche choisie
3. Support de la récupération des transactions et soldes
4. Gestion de l'authentification (login/password, potentiellement 2FA) adaptée à l'approche choisie
5. Détection et évitement des doublons lors de la synchronisation
6. Gestion robuste des erreurs avec retry et fallback si l'approche principale échoue
7. Tests unitaires avec mocks de l'API/scraping BoursoBank
8. Documentation du connecteur dans `docs/connectors/boursorama.md` avec détails de l'approche technique choisie

### Technical Implementation Details

**Recherche des options techniques** :

La première étape consiste à explorer les différentes options disponibles :

1. **API/scraping HTTP direct** :
   - Analyser les requêtes réseau lors de la connexion à l'interface web BoursoBank
   - Identifier les endpoints utilisés pour l'authentification et la récupération des données
   - Tester l'authentification avec login/password
   - Vérifier la présence de 2FA et son fonctionnement

2. **API PSD2 officielle** :
   - Rechercher la documentation officielle PSD2/DSP2 de BoursoBank
   - Vérifier la disponibilité et les conditions d'accès (certification, coûts, etc.)
   - Évaluer la facilité d'intégration

3. **Selenium/Playwright** :
   - Évaluer la complexité de l'authentification via navigateur
   - Analyser la stabilité et la maintenabilité (fragilité face aux changements d'interface)
   - Évaluer les performances et la consommation de ressources

**Format de retour de `sync_transactions()`** :

Chaque transaction doit être transformée au format standard :
```python
{
    "posted_at": datetime(...),  # Depuis dateop
    "amount": Decimal(...),      # Depuis amount (négatif pour dépenses)
    "description": str(...),     # Depuis label + comment
    "raw": dict(...)             # Métadonnées : category, counterparty, account_balance, etc.
}
```

**Détection de doublons** :

- Comparer avec les transactions existantes par :
  - Date (`posted_at`)
  - Montant (`amount`)
  - Description (`description`)
- Utiliser un identifiant unique si disponible dans les données BoursoBank
- Logger les transactions ignorées comme doublons

**Gestion des erreurs et retry** :

- Implémenter retry avec backoff exponentiel : `wait_time = base_delay * (2 ** attempt_number)`
- Max 3 tentatives pour les opérations
- Lever les exceptions appropriées après échec des retries :
  - `ConnectionTimeoutError` pour les timeouts
  - `RateLimitError` pour les rate limits
  - `AuthenticationError` pour les erreurs d'authentification

**Fallback** :

- Si l'approche principale échoue après plusieurs tentatives :
  - Essayer une approche alternative si disponible (ex: Selenium si API HTTP échoue)
  - Logger le changement d'approche
  - Lever une exception appropriée si toutes les approches échouent

**Logging** :

- Logger les opérations importantes (authentification, synchronisation, erreurs)
- Ne jamais logger les credentials, tokens, ou codes 2FA
- Utiliser le module `logging` Python avec niveaux appropriés :
  - `INFO` : Opérations réussies
  - `WARNING` : Retries, erreurs récupérables, changements d'approche
  - `ERROR` : Erreurs définitives
