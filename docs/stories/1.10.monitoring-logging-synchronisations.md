# Story 1.10: Monitoring et Logging des Synchronisations

## Status

Approved

## Story

**As a** administrateur/utilisateur,
**I want** pouvoir consulter l'historique des synchronisations et diagnostiquer les problèmes,
**so that** je peux assurer la fiabilité du système et comprendre ce qui se passe avec mes synchronisations.

## Acceptance Criteria

1. Page `/bank-connections/logs/` créée affichant l'historique des synchronisations depuis `SyncLog` :
   - Liste paginée des logs de synchronisation (ordre chronologique décroissant)
   - Affichage pour chaque log :
     - Nom du compte/connexion bancaire
     - Provider (Trade Republic, BoursoBank, Hello Bank)
     - Type de synchronisation (Manuelle/Automatique)
     - Statut (Démarrée/Succès/Erreur) avec badge coloré
     - Date/heure de début (`started_at`)
     - Date/heure de fin (`completed_at`) si disponible
     - Durée de synchronisation (calculée depuis `started_at` et `completed_at`)
     - Nombre de transactions synchronisées (`transactions_count`)
     - Message d'erreur si statut = ERROR (affiché de manière lisible)
   - Lien vers la connexion bancaire associée
   - Design cohérent avec les autres pages du dashboard
2. Filtres pour faciliter la recherche et le diagnostic :
   - Filtre par connexion bancaire (dropdown avec toutes les connexions de l'utilisateur)
   - Filtre par statut (Tous/Succès/Erreur/En cours)
   - Filtre par type de synchronisation (Tous/Manuelle/Automatique)
   - Filtre par date (date de début et date de fin)
   - Bouton "Réinitialiser les filtres"
   - Les filtres sont persistés dans l'URL pour permettre le partage/bookmark
3. Affichage détaillé des erreurs pour debugging :
   - Page de détail d'un log (`/bank-connections/logs/<log_id>/`) ou modal
   - Affichage complet du message d'erreur (`error_message`)
   - Formatage du message d'erreur pour améliorer la lisibilité (si stack trace)
   - Affichage des informations contextuelles (provider, compte, type de sync, dates)
   - Option pour copier le message d'erreur dans le presse-papier
4. Statistiques de synchronisation affichées sur la page :
   - Taux de succès global (pourcentage de synchronisations réussies)
   - Taux de succès par provider (Trade Republic, BoursoBank, Hello Bank)
   - Nombre total de synchronisations (sur la période filtrée)
   - Temps moyen de synchronisation (calculé depuis les logs avec `completed_at`)
   - Nombre total de transactions synchronisées (somme de `transactions_count`)
   - Graphique ou tableau des synchronisations par jour (optionnel, pour visualisation)
5. Alertes automatiques en cas d'échecs répétés (configurable) :
   - Détection automatique d'échecs répétés (ex: 3 échecs consécutifs pour une connexion)
   - Affichage d'un badge d'alerte sur la page des logs pour les connexions problématiques
   - Option pour configurer le seuil d'alertes (nombre d'échecs consécutifs, période)
   - Message d'alerte clair indiquant la connexion concernée et le nombre d'échecs
6. Export des logs en CSV pour analyse externe :
   - Bouton "Exporter en CSV" sur la page des logs
   - Export respecte les filtres appliqués (seulement les logs filtrés sont exportés)
   - Format CSV avec colonnes : Date début, Date fin, Connexion, Provider, Type, Statut, Durée (secondes), Transactions, Message d'erreur
   - Nom de fichier avec timestamp : `sync_logs_YYYYMMDD_HHMMSS.csv`
   - Téléchargement direct du fichier CSV
7. Rétention des logs configurable :
   - Tâche de nettoyage automatique des logs anciens (via Celery Beat)
   - Configuration de la rétention par variable d'environnement `SYNC_LOG_RETENTION_DAYS` (défaut: 30 jours)
   - Suppression automatique des logs avec `started_at` > rétention configurée
   - Logging de la suppression pour audit
   - Option pour désactiver la rétention automatique (rétention = 0 ou None)

## Integration Verification

- IV1: Les logs n'impactent pas les performances des synchronisations (requêtes optimisées avec indexes)
- IV2: Les logs sont accessibles sans exposer les credentials bancaires (aucun credential dans les logs)
- IV3: L'interface de logs est cohérente avec le reste de l'application (même design, même navigation)

**Fiabilité** :
- La tâche de nettoyage doit être robuste et ne pas supprimer accidentellement des logs récents

- La rétention doit être configurable pour permettre la désactivation si nécessaire

### Data Models Reference

[Source: `backend/finance/models.py`]

**Modèle SyncLog** :
- `bank_connection` : ForeignKey vers `BankConnection` (avec `related_name="sync_logs"`)
- `sync_type` : CharField avec choix `MANUAL` ou `AUTOMATIC`
- `status` : CharField avec choix `STARTED`, `SUCCESS`, `ERROR`
- `started_at` : DateTimeField (auto_now_add=True)
- `completed_at` : DateTimeField (nullable)
- `error_message` : TextField (blank=True)
- `transactions_count` : IntegerField (default=0)
- Indexes : `["bank_connection", "-started_at"]`, `["status"]`, `["sync_type"]`
- Ordering : `["-started_at"]`

**Modèle BankConnection** :
- `owner` : ForeignKey vers User
- `provider` : CharField avec choix (TRADE_REPUBLIC, BOURSORAMA, HELLOBANK)
- `account_name` : CharField
- Relation inverse : `sync_logs` (via `related_name`)

**Calcul de la durée** :
- Si `completed_at` est None, durée = None ou "En cours"
- Sinon, durée = `completed_at - started_at` (en secondes ou format lisible)
