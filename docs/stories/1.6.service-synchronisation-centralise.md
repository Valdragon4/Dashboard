# Story 1.6: Service de Synchronisation Centralisé

## Status

Done

## Story

**As a** développeur,
**I want** un service centralisé de synchronisation qui orchestre les connecteurs,
**so that** la logique de synchronisation est réutilisable et testable.

## Acceptance Criteria

1. Service `SyncService` créé dans `finance/services/sync_service.py`
2. Méthode `sync_account(account)` qui utilise le connecteur approprié selon le provider de `BankConnection`
3. Gestion de la détection de doublons centralisée (réutilisation de la logique existante de `_upsert_transaction`)
4. Gestion des erreurs et retry avec backoff exponentiel au niveau du service
5. Création automatique de `SyncLog` pour chaque synchronisation (début, succès, erreur)
6. Mise à jour des champs `last_sync_at` et `sync_status` sur `BankConnection` après chaque synchronisation
7. Support de la synchronisation des valorisations de portefeuille pour Trade Republic (via `sync_portfolio_valuations()`)
8. Tests unitaires du service de synchronisation avec mocks des connecteurs

## Integration Verification

- IV1: Le service peut être utilisé depuis les tâches Celery et les vues Django
- IV2: Les erreurs de synchronisation sont loggées sans casser l'application
- IV3: Les données synchronisées respectent les contraintes de la base de données existante
- IV4: La détection de doublons fonctionne correctement avec les transactions existantes

### Technical Implementation Details

**Sélection du connecteur** :

```python
from finance.connectors import (
    TradeRepublicConnector,
    BoursoBankConnector,
    HelloBankConnector,
    PLAYWRIGHT_AVAILABLE,
)

def _get_connector_for_provider(provider: str) -> BaseBankConnector:
    """Retourne le connecteur approprié selon le provider."""
    if provider == "trade_republic":
        return TradeRepublicConnector()
    elif provider == "boursorama":
        if not PLAYWRIGHT_AVAILABLE:
            raise ImportError("Playwright required for BoursoBank")
        return BoursoBankConnector()
    elif provider == "hellobank":
        if not PLAYWRIGHT_AVAILABLE:
            raise ImportError("Playwright required for Hello Bank")
        return HelloBankConnector()
    else:
        raise ValueError(f"Unknown provider: {provider}")
```

**Gestion des erreurs** :

```python
from finance.connectors.base import (
    ConnectionTimeoutError,
    RateLimitError,
    InvalidCredentialsError,
    AuthenticationError,
)

def _should_retry_error(error: Exception) -> bool:
    """Détermine si une erreur doit être retentée."""
    if isinstance(error, (ConnectionTimeoutError, RateLimitError)):
        return True
    if isinstance(error, InvalidCredentialsError):
        return False
    if isinstance(error, AuthenticationError):
        return False  # Ne pas retry les erreurs d'authentification
    return False  # Par défaut, ne pas retry
```

### Notes Techniques

**Gestion des imports conditionnels** :
- Import conditionnel des connecteurs Playwright (BoursoBank, Hello Bank) avec gestion gracieuse si Playwright n'est pas disponible.
- Variable `PLAYWRIGHT_AVAILABLE` déterminée localement dans le service.

**Gestion des erreurs** :
- Retry automatique avec backoff exponentiel uniquement pour les erreurs temporaires.
- Les erreurs définitives (credentials invalides, authentification) ne sont pas retentées.
- Toutes les erreurs sont loggées avec le module logging Python.

**Détection de doublons** :
- Priorité 1 : `raw.transaction_id` (pour Trade Republic et autres providers avec ID unique)
- Priorité 2 : `posted_at + amount + description` (fallback pour les autres providers)
- Réutilisation de la logique existante de `_upsert_transaction` depuis `loader.py` pour maintenir la cohérence.

**Transactions atomiques** :
- Utilisation de `@transaction.atomic` pour garantir la cohérence des données lors de la synchronisation.

**Compatibilité** :
- Le service peut être utilisé depuis les tâches Celery (Story 1.7) et les vues Django (Story 1.8).
- Les erreurs ne cassent pas l'application (toutes les erreurs sont catchées et loggées).
- Les données synchronisées respectent les contraintes de la base de données existante.
