### API Specifications

**API Trade Republic** :
- **Authentification** : REST API (`https://api.traderepublic.com/api/v1/auth/web/login`)
- **Données** : WebSocket (`wss://api.traderepublic.com`)
- **Format** : JSON pour REST, messages texte pour WebSocket

**Endpoints utilisés** :
- `POST /api/v1/auth/web/login` : Initie la connexion (retourne `process_id`)
- `POST /api/v1/auth/web/login/{process_id}/resend` : Renvoie le code 2FA
- `POST /api/v1/auth/web/login/{process_id}/{code}` : Vérifie le code 2FA (retourne token dans cookies)

**WebSocket Messages** :
- `connect 31 {locale_config}` : Connexion WebSocket
- `sub {message_id} {payload}` : S'abonner à un type de données
- `unsub {message_id}` : Se désabonner
- Types de données : `timelineTransactions`, `availableCash`, `portfolio`, `timelineDetailV2`

### Technical Implementation Details

**Authentification 2FA** :

Le processus d'authentification Trade Republic se fait en deux étapes :

1. **Initiation** : `initiate_login()` appelle l'API REST et retourne `process_id` et `countdown`
2. **Vérification** : `verify_2fa(code)` vérifie le code SMS et retourne le token de session

Pour l'implémentation dans `authenticate()` :
- Si `credentials` contient `2fa_code`, faire les deux étapes automatiquement
- Sinon, retourner `{"process_id": ..., "countdown": ..., "requires_2fa": True}` pour permettre l'authentification en deux étapes depuis l'interface utilisateur

**Format de retour de `sync_transactions()`** :

Chaque transaction doit être transformée au format standard :
```python
{
    "posted_at": datetime(...),  # Depuis timestamp (parser le format Trade Republic)
    "amount": Decimal(...),      # Depuis amount.value ou value (négatif pour dépenses)
    "description": str(...),     # Depuis type, title, ou description
    "raw": dict(...)             # Toutes les métadonnées originales
}
```

**Filtrage par date `since`** :

- Parser le `timestamp` de chaque transaction Trade Republic
- Comparer avec `since` si fourni
- Ne retourner que les transactions postérieures à `since`

**Gestion des erreurs et retry** :

- Implémenter retry avec backoff exponentiel : `wait_time = base_delay * (2 ** attempt_number)`
- Max 3 tentatives pour les opérations WebSocket
- Lever les exceptions appropriées après échec des retries :
  - `ConnectionTimeoutError` pour les timeouts
  - `RateLimitError` pour les rate limits (détecter via code HTTP ou message d'erreur)
  - `AuthenticationError` pour les erreurs d'authentification

**Logging** :

- Logger les opérations importantes (authentification, synchronisation, erreurs)
- Ne jamais logger les credentials, tokens, ou codes 2FA
- Utiliser le module `logging` Python avec niveaux appropriés :
  - `INFO` : Opérations réussies
  - `WARNING` : Retries, erreurs récupérables
  - `ERROR` : Erreurs définitives
