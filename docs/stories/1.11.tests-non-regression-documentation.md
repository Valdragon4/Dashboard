# Story 1.11: Tests de Non-Régression et Documentation

## Status

Approved

## Story

**As a** développeur,
**I want** des tests complets et une documentation à jour,
**so that** le système est maintenable et les nouvelles fonctionnalités sont bien intégrées.

## Acceptance Criteria

1. Tests de non-régression pour tous les imports CSV/PDF manuels existants :
   - Tests pour `import_bank_statement_from_csv()` avec tous les profils (generic, hellobank, boursorama)
   - Tests pour `import_traderepublic_from_csv()` avec différents formats CSV
   - Tests pour `import_traderepublic_pdf()` avec analyse OpenAI
   - Vérification que les imports manuels continuent de fonctionner après l'ajout de la synchronisation automatique
   - Vérification que les données importées manuellement sont compatibles avec les données synchronisées automatiquement
   - Tests avec différents formats de dates, montants, et encodages
   - Tests de gestion d'erreurs (fichiers invalides, formats incorrects, etc.)

2. Tests d'intégration pour le flux complet de synchronisation automatique :
   - Test du flux complet : création BankConnection → authentification → synchronisation → création SyncLog → création Transactions
   - Test avec les trois providers (Trade Republic, BoursoBank, Hello Bank)
   - Test de la détection de doublons entre import manuel et synchronisation automatique
   - Test de la tâche Celery `sync_all_bank_accounts()` avec plusieurs comptes
   - Test de la tâche Celery `sync_bank_account()` pour un compte spécifique
   - Test de gestion des erreurs et retry automatique
   - Test de l'authentification 2FA pour Trade Republic

3. Tests de performance pour s'assurer que le dashboard reste rapide :
   - Test de performance de la vue `dashboard` avec un grand nombre de transactions (1000+)
   - Test de performance de la vue `sync_logs_list` avec un grand nombre de logs (1000+)
   - Test de performance de la vue `accounts` avec optimisation DB (vérifier nombre de requêtes)
   - Test de performance de la vue `bank_connections_list` avec plusieurs connexions
   - Vérification que les requêtes DB restent raisonnables (< 10 requêtes pour les pages principales)
   - Test de pagination avec un grand nombre d'éléments

4. Documentation utilisateur pour la configuration des connexions bancaires :
   - Guide d'utilisation dans `docs/user-guide.md` ou `docs/user/bank-connections.md`
   - Instructions pour créer une nouvelle connexion bancaire
   - Instructions pour configurer chaque provider (Trade Republic, BoursoBank, Hello Bank)
   - Instructions pour l'authentification 2FA (Trade Republic)
   - Instructions pour activer/désactiver la synchronisation automatique
   - Instructions pour la synchronisation manuelle
   - FAQ avec problèmes courants et solutions
   - Captures d'écran ou exemples visuels (optionnel)

5. Documentation développeur pour l'ajout de nouveaux connecteurs :
   - Guide dans `docs/developer/adding-bank-connector.md`
   - Architecture des connecteurs (classe abstraite `BaseBankConnector`)
   - Étapes pour créer un nouveau connecteur :
     - Créer la classe héritant de `BaseBankConnector`
     - Implémenter les méthodes requises (`authenticate`, `sync_transactions`, `get_balance`)
     - Ajouter le provider dans `BankConnection.Provider`
     - Créer les tests unitaires
     - Documenter l'API/scraping utilisé
   - Exemples de code pour chaque étape
   - Bonnes pratiques et patterns à suivre
   - Gestion des erreurs et retry

6. Guide de migration depuis l'import manuel vers la synchronisation automatique :
   - Guide dans `docs/user/migration-manual-to-auto.md`
   - Étapes pour migrer un compte existant vers la synchronisation automatique
   - Instructions pour créer une `BankConnection` à partir d'un compte existant
   - Vérification de la compatibilité des données
   - Gestion des doublons lors de la migration
   - Recommandations pour la transition progressive

7. README mis à jour avec les nouvelles fonctionnalités :
   - Section "Synchronisation Bancaire Automatique" dans le README principal
   - Liste des providers supportés
   - Instructions d'installation et configuration
   - Variables d'environnement nécessaires (`ENCRYPTION_KEY`, `SYNC_LOG_RETENTION_DAYS`, etc.)
   - Configuration Celery Beat pour la synchronisation automatique
   - Liens vers la documentation utilisateur et développeur

### Technical Constraints

[Source: `docs/prd.md#Technical Constraints`]

**Performance** :
- Les tests de performance doivent être rapides (< 5 secondes par test)
- Utiliser des fixtures pour créer les données de test
- Nettoyer les données après les tests

**Sécurité** :
- Ne jamais exposer de credentials réels dans les tests
- Utiliser des mocks pour les APIs externes
- Utiliser des données de test fictives

**Fiabilité** :
- Les tests doivent être reproductibles
- Éviter les dépendances sur l'état externe (APIs, fichiers, etc.)
- Utiliser des mocks et des fixtures appropriés

### Testing Requirements

**Couverture de tests** :
- Objectif : maintenir ou améliorer la couverture existante
- Mesurer avec `coverage.py`
- Couvrir les cas principaux et les cas limites

**Types de tests** :
- Tests unitaires : fonctions individuelles isolées
- Tests d'intégration : flux complets avec mocks
- Tests de performance : mesures de temps et requêtes DB

**Outils** :
- `coverage.py` pour mesurer la couverture
- `django.test.TestCase` pour les tests Django
- `unittest.mock` pour les mocks
